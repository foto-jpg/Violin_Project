Violin Practice Evaluator - คู่มืออธิบายระบบ

เอกสารนี้อธิบายการทำงานของระบบทั้งหมด แยกตามฟังก์ชันหลักและย่อย พร้อมดึงโค้ดสำคัญมาประกอบ และวิธีที่ไฟล์ต่าง ๆ เรียกใช้กัน

1. ระบบนี้คืออะไร

เว็บแอปช่วย ตรวจการเล่นไวโอลินอัตโนมัติ รับ input 2 ทาง แล้วนำมาเทียบกัน:

ภาพโน้ตเพลง โน้ต (OMR)

อัปรูป/PDF โน้ตเพลง ระบบใช้ OMR (Optical Music Recognition) แปลงเป็น MusicXML แล้วถอดเป็นรายการโน้ต ทำได้ 2 เครื่องยนต์:

  • oemer - deep learning (ONNX, ใช้ GPU)
  • Audiveris - rule-based (Java)

คลิปเสียง pitch (Audio)

อัปไฟล์เสียงที่เล่นจริง ระบบใช้ CREPE (fine-tuned สำหรับไวโอลิน) ถอด pitch ราย frame แล้วแบ่งเป็นโน้ต (onset/rhythm ด้วย librosa)

จากนั้น Match เอาเสียงไปจูนกับโน้ตด้วย DTW alignment แล้วบอกว่าโน้ตแต่ละตัว เล่นถูก / เพี้ยน / ตก

มี 2 หน้าตา (UI variant) จากโค้ดชุดเดียวกัน:
• Local (research) - แท็บ "ภาพ" เปรียบเทียบ oemer Audiveris บนภาพเดียวกัน + แท็บ "เสียง" (ใช้ GPU)
• HF (deploy) - pipeline เต็ม: Audiveris + CREPE + Match + Synced Score Player (CPU เท่านั้น, ปิด oemer)
สลับด้วย build flag NEXT_PUBLIC_HF=1

5. lib/api.ts - ชั้นเรียก API + Proxy Frontend

ทุกการเรียก backend ผ่านไฟล์เดียว lib/api.ts ซึ่งเลือก base URL ตาม environment:

const isHF = process.env.NEXT_PUBLIC_HF === '1'
// local  ยิงผ่าน proxy ของ Next.js ; HF  same-origin ตรงเข้า FastAPI
const BASE = process.env.NEXT_PUBLIC_API_BASE ?? (isHF ? '/api' : '/api/backend')
export const API_BASE = BASE        // ให้ <audio> / OSMD ใช้

export async function submitJob(file, engine, tempo=120) {       // OMR
  const form = new FormData(); form.append('file', file)
  form.append('engine', engine); form.append('tempo', String(tempo))
  const res = await fetch(`${BASE}/omr/process`, {method:'POST', body:form})
  return res.json()                 //  { job_id, status }
}
export async function getResult(jobId)      // poll OMR
export async function submitAudio(file, tempo)   // CREPE
export async function startMatch(omrId, audioId, tempo)  //  job_id (async)
export async function getMatchResult(jobId)

Proxy (local เท่านั้น)

app/api/backend/[...path]/route.ts เป็น Next.js Route Handler ที่ forward /api/backend/<path> BACKEND_URL/api/<path> (เพราะ local frontend(:3100) คนละ origin กับ backend(:8300))

บน HF ใช้ output:'export' (static) จึงไม่มี proxy - Dockerfile จะ rm -rf app/api ก่อน build แล้ว FastAPI เสิร์ฟ /api/* เองที่ origin เดียวกัน

7. Synced Score Player (OSMD) Frontend

SyncedScorePlayer.tsx เรนเดอร์โน้ตจริงด้วย OpenSheetMusicDisplay แล้ว sync กับเสียงที่เล่น

1) โหลด + เก็บลิสต์โน้ตตามลำดับ

await osmd.load(xml); osmd.render()
// เดินทั้ง graphical sheet เก็บ "หัวโน้ต" (ข้าม rest) เรียงตามลำดับ
for (const staves of osmd.GraphicalMusicSheet.MeasureList)
  for (const measure of staves)
    for (const se of measure.staffEntries)
      for (const gve of se.graphicalVoiceEntries)
        for (const gn of gve.notes)
          if (!gn.sourceNote.isRest()) gNotes.push(gn)

function paint(gNote, color) {            // แตะ <g> ของหัวโน้ตตรง ๆ
  const el = gNote.getSVGGElement()
  el.querySelectorAll('path, ellipse, rect').forEach(p => p.setAttribute('fill', color))

10. Image preprocessing (จุดสำคัญที่สุดของความแม่น) Backend

utils/files.py save_upload() เรียก 3 ขั้นตามลำดับ บนไฟล์ที่ทั้ง 2 engine ใช้ร่วมกัน:

def save_upload(content, mime, uploads_dir):
    ... เขียนไฟล์ ...
    auto_orient(out_path)      # 1) หมุนให้ตั้งตรง
    trim_whitespace(out_path)  # 2) ตัดขอบว่าง
    upscale_for_omr(out_path)  # 3) ขยายถ้าความละเอียดต่ำ

10.1 auto_orient - แก้รูปถ่ายมือถือหมุน/กลับหัว

ตรวจว่าเส้น staff เป็นแนวตั้งไหม โดยเทียบ "ความเป็นคาบ" (autocorrelation peak) ของ projection แนวแถว vs คอลัมน์ ถ้าคอลัมน์เป็นคาบชัดกว่า = staff ตั้ง หมุน 90° แล้วใช้ Tesseract OSD ตัดสินหัว/ท้าย (พลิก 180°)

row_score = _autocorr_peak(dark.sum(axis=1))   # คาบถ้า staff "นอน"
col_score = _autocorr_peak(dark.sum(axis=0))   # คาบถ้า staff "ตั้ง"
if col_score > 0.3 and col_score > row_score*1.3:
    rgb = rgb.transpose(ROTATE_270)            # หมุน 90°
rot = _osd_rotation(rgb)                        # tesseract --psm 0
if rot in (90,180,270): rgb = rgb.rotate(-rot)  # แก้กลับหัว

10.2 trim_whitespace - แก้ oemer พังกับหน้าโล่ง

ตัดเหลือเฉพาะ bounding box ของเนื้อหา (+ขอบ 40px) เพราะ oemer หา staff ไม่เจอเมื่อโน้ตอยู่นิดเดียวกลางหน้าโล่ง

10.3 upscale_for_omr - แก้ Audiveris ตีว่า resolution ต่ำ

วัด interline (ระยะห่างเส้น staff) ด้วย autocorrelation แล้ว ขยายเฉพาะเมื่อต่ำกว่า 13px ให้ถึง ~20px (ถ้าพอแล้วไม่แตะ - เพราะขยายเกินทำเส้นแตก)

interline = _estimate_interline(gray)     # autocorr peak ของ projection แนวแถว
if interline >= 13: return                 # พอแล้ว - อย่าแตะ
factor = min(20/interline, 4.0)
rgb.resize((w*factor, h*factor), LANCZOS)
ผลจริงที่ทดสอบ: หน้า Gavotte (interline 9 ขยาย 130 โน้ต) และ Twinkle (interline 14 ไม่ขยาย 14 โน้ต) ทำงานทั้งคู่

12.2 align_audio_to_score - แก่นการให้คะแนน

ใช้ chroma ของเสียง (จาก CREPE) เทียบ chroma ที่สังเคราะห์จากโน้ตที่คาดหวัง ด้วย banded DTW แล้ว map ช่วงเวลาเสียงให้แต่ละโน้ต วัด pitch ที่เล่นจริง จัดสถานะ:

def _classify(cents, voiced):
    if voiced < 0.3:        return "missed"        # ไม่มีเสียง
    if abs(cents) > 50:     return "wrong_pitch"   # เพี้ยน > ครึ่งเสียง
    return "correct"                               # ถูก
# คืน { expected_count, aligned_count, summary{correct,wrong_pitch,missed},
# notes:[{index, expected_name, start_sec, end_sec, status, ...}] }

ใช้ DTW บน chroma แทนการ segment โน้ตจากเสียงตรง ๆ เลี่ยงปัญหา note-segmentation และจัดให้ตรงกับโน้ตที่อัปโหลด

_store: dict[str, JobState] = {}
_lock = threading.Lock()
def is_any_running() -> bool:
    return any(j.status == JobStatus.running for j in _store.values())
